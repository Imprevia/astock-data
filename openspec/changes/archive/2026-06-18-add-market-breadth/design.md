## Context

`astock-data` 当前公开面是严格的个股数据门面：18 个公开函数中，17 个是 `get_*` 数据入口，另一个是 `resolve_ticker` 安全边界。多数服务函数都以 `symbol` 为输入，并最终依赖 `resolve_ticker` 将用户输入收敛到合法 A 股个股 `Ticker`。

市场广度数据不是个股数据。指数快照、涨跌停家数和连板梯队都属于盘后复盘的「市场层面」事实数据，不应强行塞进 `resolve_ticker`。尤其指数存在语义冲突：`000001` 在个股 resolver 中是平安银行，而在指数语境中常被用户理解为上证指数。为了避免安全边界被污染，本变更必须为市场层数据提供独立入口与独立数据映射。

前期 spike 已验证：

- 东财 `push2` `stock/get` 可通过写死 secid 获取指数快照（上证、深证、创业板、科创50、沪深300、中证500均可用）。
- 东财 `push2` `clist/get` 可拉取全 A 实时行情，按涨跌幅阈值本地统计涨停/跌停家数。
- 同花顺 `getharden` 只返回涨停热股字段全集，不包含连板数。
- 东财 `push2ex` 涨停/跌停/连板专题接口在无特殊 token 时返回 `rc=102` 或 404，不可作为公开稳定数据源。
- Tushare `limit_list_d` 理论上提供 `limit_times` 连板数，但需要额外积分权限；实测给定 token 有效但无该接口权限。

因此连板梯队采用无状态派生计算：先确定当日涨停股，再对每只涨停股回溯历史 K 线，连续计算涨停天数。

## Goals / Non-Goals

**Goals:**

- 新增单一聚合入口 `get_market_breadth()`，一次返回指数快照、涨跌停家数与连板梯队。
- 将指数输入面限制为内部枚举 secid，不允许用户传任意指数代码。
- 保持 `resolve_ticker` 的个股安全边界不变。
- 连板梯队使用无状态现算，不新增 SQLite 状态库、定时任务或跨日状态机。
- 继续保持 Python API、CLI、MCP 三个公开面同步。
- 所有返回值仍为 Pydantic 结构化模型，不返回自由文本报告。
- 所有东财请求仍统一经过 `EastmoneyClient` 的 chokepoint、限流与错误映射。

**Non-Goals:**

- 不引入 Tushare 依赖、token 配置或付费权限前提。
- 不实现持久化连板状态库（例如 `zt_state.sqlite`）。
- 不实现盘中炸板家数统计；当前数据源无法稳定提供盘中触板/炸板事实。
- 不提供投资建议、情绪评分、交易信号或复盘结论。
- 不改变现有 18 个公开入口的签名或行为。
- 不扩展 `resolve_ticker` 支持指数代码。

## Decisions

### 1. 使用单一聚合入口，而不是三个独立入口

新增 `get_market_breadth(date: str = "") -> MarketBreadthResult`。

理由：复盘场景通常同时需要指数、涨跌停家数和连板梯队；一次聚合返回比三个松散函数更契合 `daily-review` 的「看大局」流程。拆成 `get_indices()`、`get_limit_counts()`、`get_board_ladders()` 会把数据源延迟、warnings 和日期语义拆散，调用方还要自行组合。

备选方案：三个独立公开入口。拒绝原因是公开入口膨胀更大，且不符合用户确认的「一次拿全」场景。

### 2. 指数 secid 写死成枚举

默认指数集合：

- `sh`: 上证指数，`1.000001`
- `sz`: 深证成指，`0.399001`
- `cyb`: 创业板指，`0.399006`
- `kc50`: 科创50，`1.000688`
- `hs300`: 沪深300，`1.000300`
- `zz500`: 中证500，`1.000905`

理由：指数不是个股，不应进入 `resolve_ticker`。写死枚举避免用户输入歧义，也避免指数代码与个股代码重名。

备选方案：新增 `resolve_index()`。拒绝原因是当前需求不需要开放任意指数输入；枚举更小、更安全、更可测。

### 3. 涨跌停家数通过东财 clist 全量行情本地分类

`EastmoneyClient` 增加 `clist` 翻页辅助能力，服务层拉取全 A 行情后根据代码板块判定涨跌停：

- 主板：`±9.8%`
- 创业板 / 科创板：`±19.5%`
- 北交所：`±29.5%`
- ST：`±4.8%`（需要基于名称包含 `ST` / `*ST` 判定）

理由：公开可达的东财涨跌停专题接口不可用；`clist` 是已验证可达的数据源。该口径是收盘涨跌幅近似，能满足盘后复盘的市场广度需求。

备选方案：使用东财 `push2ex` 涨跌停池。拒绝原因是 spike 中稳定返回 `rc=102` 或 404，缺少公开可复现调用条件。

### 4. 连板梯队采用无状态 K 线回溯现算

流程：

1. 从 clist 分类得到当日涨停股票集合。
2. 对每只涨停股调用现有 K 线管道，拉取 `date - lookback_days` 到 `date` 的日 K。
3. 从目标日期向前倒序，按对应板块阈值判断连续涨停天数。
4. 将结果按 `boards` 分组形成 `board_ladders: dict[int, list[BoardItem]]`。

默认 `lookback_days = 20`，用于覆盖绝大多数连板高度。该计算不持久化跨日状态，只依赖现有 `CsvKlineCache` 提升性能。

理由：持久化连板状态会引入外部状态机，违反 `astock_data` 纯数据层边界。无状态现算与现有 `get_indicators` 的派生指标计算同构：都是从历史行情即时派生，不维护业务状态。

备选方案：维护 `zt_state.sqlite` 增量累加。拒绝原因是需要定时维护和状态腐烂处理，属于应用层状态机，不属于数据层。

### 5. 不承诺炸板家数

`LimitStats` 不包含 `broke_board_count`，或仅在未来数据源可稳定提供时另行新增。

理由：炸板是盘中触板后未封住的事实，需要盘中明细或专题接口。当前可用的 `clist` / K 线只能看收盘涨跌幅，无法可靠判断「盘中曾涨停后炸板」。将其放入模型会制造虚假精度。

### 6. 模型放在 `models/market.py`，服务放在新模块 `services/market_breadth.py`

模型仍属于行情/市场数据契约，放入 `models/market.py` 可避免新增小文件扩散。服务编排跨指数、clist、K 线回溯，复杂度高于 `market_data.py` 的 K 线/指标职责，因此新增 `services/market_breadth.py` 更清晰。

### 7. `source` 保持字符串，细分来源放入 `raw` 或 warnings

`ResultBase.source` 当前为 `str`。本变更不修改基类类型，避免影响所有现有模型。`MarketBreadthResult.source` 使用一个稳定的汇总值（例如 `eastmoney+derived`），并在 `raw` 中保留更细的来源拆分。

备选方案：将 `source` 升级为结构化对象。拒绝原因是这会影响现有所有结果模型、格式化器与测试断言，代价过高。

## Risks / Trade-offs

- [风险] 连板为无状态现算，口径可能与行情软件差 1 板 → [缓解] 在模型 `description` 与 `warnings` 明示派生口径、阈值和 lookback 默认值。
- [风险] `clist` 全量翻页可能较慢且触发东财限流 → [缓解] 复用 `EastmoneyClient` 的统一节流，并尽量依赖现有缓存热路径。
- [风险] 指数枚举可能不够全 → [缓解] 先覆盖复盘高频指数，后续可通过 proposal 扩展枚举。
- [风险] 现有 `source` 为字符串，无法表达混合数据源 → [缓解] 保持 `source` 为汇总字符串，细分来源放入 `raw`。
- [风险] 没有炸板家数会削弱市场情绪完整度 → [缓解] 这次不承诺该字段，避免伪精度；后续若出现稳定来源再补。

## Migration Plan

1. 新增模型与服务实现，保持现有 API 不变直至新入口稳定。
2. 在 `api.py`、`services/__init__.py`、CLI、MCP 中注册新入口，确保 Python/CLI/MCP 同步。
3. 更新 README 与测试，调整公开入口计数从 18 到 19。
4. 先用离线 mock 测试验证结构，再用真实调用做 smoke。
5. 若新入口验证失败，回滚只需撤销 `market-breadth` 相关文件与导出，不影响现有 18 个入口。
